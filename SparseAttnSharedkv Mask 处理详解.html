SparseAttnSharedkv 的 Mask 处理详解

这个 TileLang 算子是怎么决定「当前这个词能看见哪些词、不能看见哪些词」的 —— 用一个小例子从头讲到尾，高中生也能读懂

本文配套代码：sparse_attn_sharedkv_tilelang/kernel.py，重点是 T.Scope("V") 段里第 393–550 行构建并施加 mask 的部分。

目标读者：没学过神经网络的高中生。你只需要会这几样：

  • 加减乘除、比大小（≤、<）、求最大值；
  • 知道指数函数 exp(x)（也写作 eˣ）：x 越大它越大，且永远是正数；特别地 exp(−∞) = 0；
  • 知道「整数除法」//：例如 5 // 2 = 2（只取商、丢掉余数）。

全文用同一个小例子贯穿，每一步都能拿计算器自己验算。涉及在线 softmax 的细节可参考姊妹篇 《Online Softmax 原理详解》。

0先说人话：mask 到底是什么、为什么需要它

想象你在读一个句子，读到第 6 个字（我们叫它 query，"提问的人"）。模型要做的事是：让这个字去「看」句子里的其他字（这些字叫 key，"被看的人"），算出它该多关注每一个字，再据此把信息汇总起来。

但不是每个字都允许被看。比如：

  • 不能看未来。读到第 6 个字时，第 7、8 个字还没出现，偷看就作弊了（这叫因果 causal 限制）。
  • 不能看不存在的东西。有些位置只是为了凑整数而留的空位（占位符），里面没有真东西。
  • 只看该看的。这个算子很「省」，只让 query 看最近的一小段，外加远处被特意挑出来的几个。其余一律不看。

Mask（掩码）就是这样一张「能看 / 不能看」的名单。它对每一个 key 给一个标记：✓ 能看 或 ✗ 不能看。算子的任务，就是正确地造出这张名单，并让"不能看"的 key 对结果毫无影响。

本文要回答的核心问题就是：这个算子怎么造出这张名单，又怎么让"✗"的 key 真正变成"零贡献"。下面先立一个小例子。

1立一个贯穿全文的小例子

真实算子里的数字很大（一次看 64 个 token、每个 token 是 512 维向量、有 64 个"注意力头"）—— 这些大数字不影响 mask 的逻辑，只会让手算变难。所以本文把它们等比例缩小成能用手算的小数字；逻辑和真实代码一模一样。

ori_win_left滑动窗口往左能看多远7127

NI_ori窗口要分成几块 = ⌈窗口/BI⌉22

为什么可以忽略"头"和"512 维"？因为 mask 只跟位置有关（第几个 query 看第几个 key），对所有 64 个头、所有 512 维都一模一样。所以代码只造一份长度为 BI 的 mask，然后套用到所有头上。本文只盯着位置，头和维度先放一边。

例子的舞台：8 个原始 token，4 个压缩 token，聚光灯打在位置 5 的 query 上

这条序列有 8 个原始 token，位置编号 0~7。把每 cmp_ratio = 2 个原始 token 打包"压缩"成 1 个，就得到 4 个压缩 token（编号 0~3）：压缩 token 0 概括原始 [0,1]，压缩 token 1 概括 [2,3]，依此类推。

我们全程盯着位置 5 的那个 query（记 s = 5）。它是一个还在句子中间的词 —— 这样才能同时看到「因为是中间，所以右边有东西要遮」的效果。

这个算子对每个 query 分两趟（pass）来看 key：

  • ori pass 滑动窗口：看最近的一段原始 token（窗口大小 8，分 2 块）。负责"近处细节"。
  • cmp pass 压缩稀疏：看远处被挑出来的压缩 token（最多 8 个，分 2 块）。负责"远处大意"。

两趟各有自己的 mask 规则。我们先讲清楚「mask 是怎么起作用的」这件公共机制，再分别看两趟。

2mask 的本质：给分数加 −∞

query 看每个 key，会先算出一个分数（score），分数越高表示"越该关注"。之后这些分数会过一道叫 softmax 的工序，变成一组加起来等于 1 的权重。softmax 的关键一步是对每个分数取 exp（指数）：

权重∝exp(分数)，且exp(−∞)=0

这就给了我们一个极其干净的遮挡办法：

要遮住一个 key，就在它的分数上加一个 −∞（负无穷）。

  • 能看的 key：分数 + 0 → 分数不变 → 权重正常。
  • 被遮的 key：分数 + (−∞) → 变成 −∞ → exp(−∞) = 0 → 权重正好是 0，对结果毫无影响。

这叫 additive mask（加性掩码）。妙处在于：不管那个 key 的原始分数多高，加上 −∞ 后权重一定归零，干净利落、没有例外。

所以代码里造 mask 的套路永远是三步，后面两趟 pass 都照这个套路走：

  1. 判断每个 key 能不能看 → 得到一串 0/1 的标记（叫 mask_ub，1=能看，0=遮住）。
  2. 翻译成加性掩码：能看→加 0，遮住→加 −∞（代码里用 T.tile.select 一步完成）。
  3. 相加：把这个加性掩码加到分数上，再交给 softmax。

代码里的小细节：算子其实是先把加性掩码（0 / −∞）放进 acc_s_ub，再 加上原始分数、最后乘以缩放系数 scale。因为 (−∞ + 任何数) × 正数 = −∞，顺序不影响结果：被遮的位置最终还是 −∞。见第 513–549 行。

3第一种 mask：滑动窗口的因果遮挡（ori pass）

ori pass 让 query 看最近的一段连续原始 token。"最近一段"就是一个长度为 8 的滑动窗口，窗口右端顶着 query 自己，左端往回数 7 个：

窗口=[ max(s−7, 0) , s ]，本例 s=5→[ max(−2,0), 5 ] = [0, 5]

注意右端是 s = 5（含自己），不是 7。位置 6、7 是未来，不能看。这就是因果遮挡。

关键观察：左边界天然成立，只需检查右边界

窗口要分成 NI_ori = 2 块，每块 BI = 4 个。每块的起点是 chunk_start = ori_left + 块号 × 4，而 ori_left = max(s−7,0) = 0。于是：

  • 第 0 块：token [0, 1, 2, 3]
  • 第 1 块：token [4, 5, 6, 7]

因为每块都从 ori_left 往右排，所以不可能有 token 比左边界还小 —— 左边界自动满足。代码因此只检查右边界一个条件：

能看 ⟺ token 编号 ≤ s（本例 ≤ 5）

套到两块上，mask 名单是这样（绿=能看，红=被 −∞ 遮住）：

窗口怎么"滑动"？换一个 query 位置，s 变了，窗口 [max(s−7,0), s] 整段跟着平移。比如 query 在位置 10：窗口 = [3,10]，看的是 token 3~10。窗口像一个固定宽度的取景框，跟着 query 一起往右滑 —— 这正是"滑动窗口 sliding window"名字的由来。只有当 query 还在句子很靠前（s < 7）时，左端才会被 max(...,0) 顶在 0，窗口右侧才会出现"够不着、被遮住"的格子（就像本例的 6、7）。

4第二种 mask：压缩稀疏的双重遮挡（cmp pass）

cmp pass 负责"远处大意"。它不看原始 token，而看压缩 token：每 cmp_ratio = 2 个原始 token 概括成 1 个压缩 token。本例 8 个原始 → 4 个压缩（c0~c3）。

4.1 压缩空间里的"不能看未来"：threshold

query 在原始空间能看到位置 0..5。换算到压缩空间，它能看几个压缩 token？答案由一个门槛 threshold 决定：

threshold=s + 1cmp_ratio（整除）=5 + 12=3

意思是：压缩 token 编号 < 3 才能看，即 c0、c1、c2 能看，c3 不能看。

为什么这个门槛是对的？c3 概括的是原始 token 6、7 —— 正是 query 看不到的未来！所以"压缩空间 idx < 3"和"原始空间不能看 6、7"说的是同一件事。两套 mask 在边界上完全咬合，绝不矛盾。

4.2 还要防"占位符"：idx ≥ 0

cmp pass 看的是"被特意挑出来的"压缩 token，编号存在一个列表 cmp_indices 里。但列表长度固定为 K=8，挑不满时用 −1 补齐（占位符，表示"这格没有真东西"）。所以还要第二个条件：编号必须 ≥ 0（≥0 才是真编号，−1 是占位符要遮掉）。

合起来，cmp pass 的 mask 是两个条件同时成立（逻辑 AND）：

能看 ⟺ ( idx ≥ 0 ) 且 ( idx < threshold )

5双重保险：搬数据时也按 mask 填 0

到这里 mask 的"数学"已经讲完了：被遮的 key 分数加 −∞，权重归零。但代码还多做了一件事 —— 在搬运 KV 数据时，也按 mask 把非法位置填成 0。为什么要画蛇添足？因为有内存安全问题：

  • 一个 −1 占位符，如果硬要拿它当编号去内存里取数据，会读到非法地址（越界、读到垃圾，甚至崩溃）。
  • 所以代码在取数前先看一眼：编号合法（≥ 0，或 ori 的 ≤ s）才真去取；否则直接往工作区填一行 0，根本不碰内存。
# kernel.py 第 413-428 行（ori）/ 第 489-503 行（cmp）：取数前的安全闸门
if g_idx <= ori_right:        # ori：编号没越过当前位置才取
    ... 解析分页表，T.copy(真实 KV 行 → kv_ub)
else:
    T.tile.fill(kv_ub, 0.0)    # 否则填 0，不碰内存

if cmp_idx >= 0:              # cmp：是真编号才取（-1 不取）
    ... 解析分页表，T.copy(真实 KV 行 → kv_ub)
else:
    T.tile.fill(kv_ub, 0.0)    # -1 占位符 → 填 0

一句话记住：mask 被执行了两遍 —— 一遍在搬数据时（防止读非法内存，填 0 占位），一遍在算分数时（加 −∞ 让权重归零）。前者保平安，后者保正确。

6s_global 的玄机：query 比 kv 短怎么办

前面一直用 s 表示 query 的位置。代码里这个位置其实叫 s_global，它由一条公式算出：

s_global=act_kv−act_q+s_i

本文的主例里 act_kv = act_q = 8，所以 s_global = s_i（位置就是序号），公式看不出名堂。但在另一种常见情形下它很关键：KV 缓存里已经有一批老 token，这次只算少数几个新 query（比如聊天时模型逐字往外蹦，叫 decode；或者分批喂入 prompt）。

举例：内存里已存了 act_kv = 8 个 token（位置 0~7），这次只来了 act_q = 1 个新 query（s_i = 0）。它其实是第 8 个词，真实位置应该是 7：

s_global = 8 − 1 + 0 = 7（不是 0！）

所以前面所有用到 s 的地方（窗口右端、threshold），严格说都是 s_global。理解了它，你就明白为什么这套 mask 不管"一次算多少 query"都成立。

7永远不被 mask 的虚拟 token：sink

还有一个特殊的"key"叫 sink（直译"水槽/汇点"）。它不是序列里的真 token，而是一个虚拟的、永远在场、永远不被 mask 的占位 key，每个注意力头一个。它的作用是给 softmax 一个"谁都不特别想看"时的泄洪口：当所有真 key 的分数都不高时，注意力可以"倒"一部分到 sink 上，相当于一个可学习的"什么都不关注"选项。

实现上，sink 不走 mask 这条路，而是被当作初始状态"种"进在线 softmax里：把 (行最大值, 行求和) 初始化为 (sink_h, 1.0)，等价于"先放进去一个分数为 sink_h、对应 value 全为 0 的虚拟 key"。因为它的 value 是 0，所以它只改变分母（让权重总和略小于把它去掉时的值），不给输出加任何内容。

本文聚焦 mask，sink 只点到为止。它的完整推导（为什么种子是 (sink, 1)、怎么不影响数值）见姊妹篇 《Online Softmax 原理详解》第 6 节。对应代码在 kernel.py 第 382–387 行（T.copy(Sinks..., m_i) 把 sink 种进 m_i，并把 sumexp 填 1.0）。

于是这个 query 实际"看见"的，是 原始 token 0~5（近处细节） + 压缩 token c0、c1、c2（远处大意） + sink（兜底）。所有被遮的格子（未来的 6、7；占位 −1）权重都是 0。softmax 只在这些"✓"上分配权重、加权汇总出输出 —— 这正是开头那张图说的事。

真实算子里只是把数字放大：窗口 128 个原始 token（2 块×64）+ 最多 512 个压缩 token（8 块×64）+ sink，对 64 个头同时算。判断规则一字不差。

10一页速查 + 总结

对每个 query（真实位置 s = s_global），mask 决定它能看哪些 key：

# ===== ori pass（滑动窗口·近处）：窗口 [max(s-7,0), s]，分 2 块 =====
for 块 in ori 的 2 个块:
    生成本块编号 idx = [chunk_start, chunk_start+1, ...]
    mask = ( idx ≤ s )                 # 只查右边界；左边界由起点保证

# ===== cmp pass（压缩稀疏·远处）：阈值 threshold = (s+1)//cmp_ratio =====
for 块 in cmp 的若干块:
    idx = 外部稀疏列表（SCFA） 或 稠密 0,1,2,...（CFA）
    mask = ( idx ≥ 0 ) and ( idx < threshold )  # 去占位符 且 不越界

# ===== 公共：把 mask 变成加性掩码，让被遮的权重归零 =====
加性掩码 = where(mask, 0.0, -inf)      # 能看→0，被遮→-inf
分数 = (分数 + 加性掩码) × scale            # 被遮的位置 → -inf
权重 = softmax(分数)                       # exp(-inf)=0 → 被遮零贡献
# 另：取 KV 数据时，非法编号直接填 0（内存安全，双重保险）
# 另：sink 作为永不 mask 的虚拟 key，种进 softmax 初始状态

三句话总结这个算子的 mask：

  1. 本质：mask = 给被遮 key 的分数加 −∞，靠 exp(−∞)=0 让它权重归零（additive mask）。规则对所有头、所有维度通用，只跟位置有关。
  2. 两类规则：① ori 滑窗只查 idx ≤ s（因果，右边界）；② cmp 压缩查 idx ≥ 0（去 −1 占位）且 idx < threshold（压缩空间因果）。两套规则在"看不到未来"这件事上完全咬合。
  3. 两道执行：搬数据时按 mask 填 0（保不崩），算分数时加 −∞（保不错）；外加一个永不 mask 的 sink 兜底。
