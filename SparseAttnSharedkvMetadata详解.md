# SparseAttnSharedkvMetadata 算子详解

> 这份文档不预设你懂神经网络、不预设你写过 numpy。
> 我们会从「这个算子在做什么」开始，一步步把它讲透。

---

## 0. 一句话回答：这个算子到底做什么？

**它不做任何数学计算，它只做一件事：分工。**

它的工作就像一个**工头**：手下有 24 个工人（NPU 上的 24 个 AI Core），
面前有一堆「拼图块」要拼。它的任务是 —— 决定**哪个工人拼哪几块**，
让所有人差不多同时完工（不要有人累死、有人闲死）。

它把这个分工方案写成一张表（叫 `metadata`），交给真正干活的算子
`SparseAttnSharedkv` 去执行。

```
       ┌──────────────────────────────────────────────┐
       │  SparseAttnSharedkvMetadata（本算子）          │
       │  ─────────────────────────                   │
       │  输入: 每个 batch 的实际长度                   │
       │  工作: 估算每块的开销 + 分配给 24 个核         │
       │  输出: metadata（一张 1024 个 int32 的表）     │
       └────────────────────┬─────────────────────────┘
                            │ metadata
                            ▼
       ┌──────────────────────────────────────────────┐
       │  SparseAttnSharedkv（下游算子，真正算 Attn）   │
       │  ─────────────────────────                   │
       │  按 metadata 的分工执行 Attention 计算         │
       └──────────────────────────────────────────────┘
```

## 1. 一点点必要的背景

### 1.1 什么是「Attention」？

想象你在读一本书，读到第 100 页时，你的大脑会**回头去看**前面相关的内容，
来理解当前这一页的意思。神经网络中的 Attention 做的就是这件事。

具体到数字层面，Attention 涉及三个矩阵：

```
   Q (Query, 查询)    K (Key, 键)       V (Value, 值)
   ┌─────────┐        ┌─────────┐       ┌─────────┐
   │ ▓ ▓ ▓ ▓ │        │ ░ ░ ░ ░ │       │ ░ ░ ░ ░ │
   │ ▓ ▓ ▓ ▓ │        │ ░ ░ ░ ░ │       │ ░ ░ ░ ░ │
   │ ▓ ▓ ▓ ▓ │        │ ░ ░ ░ ░ │       │ ░ ░ ░ ░ │
   │ ▓ ▓ ▓ ▓ │        │ ░ ░ ░ ░ │       │ ░ ░ ░ ░ │
   └─────────┘        └─────────┘       └─────────┘
   「我现在想问什么」   「目录是什么」     「实际内容是什么」
```

核心运算是：

```
              ┌─────────────────────────────┐
   Output  =  │  softmax(Q × Kᵀ × scale)  │  ×  V
              └─────────────────────────────┘
                          ▲
                          │
              一张 S1 × S2 的「注意力分数表」
              S1 = Q 有多少个 token
              S2 = K 有多少个 token
```

其中 `Q × Kᵀ` 会算出一个**大矩阵**，告诉你 Q 的每一个 token 应该「关注」K 的哪些 token。
这张矩阵是 Attention 的核心，也是计算量的主要来源。

### 1.2 「稀疏」是什么意思？

完整的 Attention 是 Q 的每个 token 都看 K 的**全部** token：

```
        K 的全部 token (S2 = 8)
        ──────────────────────►
      ┌─┬─┬─┬─┬─┬─┬─┬─┐
   ▲  │█│█│█│█│█│█│█│█│  ← token 0 看了所有 K
   │  ├─┼─┼─┼─┼─┼─┼─┼─┤
   Q  │█│█│█│█│█│█│█│█│  ← token 1 看了所有 K
   的 ├─┼─┼─┼─┼─┼─┼─┼─┤
   每 │█│█│█│█│█│█│█│█│
   个 ├─┼─┼─┼─┼─┼─┼─┼─┤
   ↓  │█│█│█│█│█│█│█│█│
      └─┴─┴─┴─┴─┴─┴─┴─┘
        计算量 = S1 × S2，序列变长时会爆炸增长
```

**稀疏 Attention** 的思路是：不必每个 token 都看所有 K，只看一部分就够了。
这样可以省下大量算力。

本算子配合的下游算子（`SparseAttnSharedkv`）支持两种稀疏方式：

- **Sliding Window（滑动窗口，简称 Win）**：每个 Q token 只看最近的 128 个 K token
- **Compressed / Sparse Compressed（压缩，简称 Cmp）**：把 K 压缩成更短的序列再算

可以同时启用：

```
        ←── Cmp 部分 ──→  ←── Win 部分 (128) ──→
      ┌─────────────────┬──────────────────┐
   ▲  │  ░░░ 看压缩版 ░░│  ███ 看原始版 ███│  ← Q token 当前位置
   │  ├─────────────────┼──────────────────┤
   Q  │  ░░░░░░░░░░░░░░░│      ██████████  │  ← 注意 Win 在「右下」移动
   ↓  ├─────────────────┼──────────────────┤
      │  ░░░░░░░░░░░░░░░│          ██████  │
      └─────────────────┴──────────────────┘
```

每个 Q token 要算的内容 = 「Win 部分」+「Cmp 部分」。
这两部分的大小**按 Q 的位置变化**——这正是分工困难的核心原因。

具体来说有 **3 个变化源**。

> **下文中的"行"** 指代码里的 **S1G 行** —— 它代表 **1 个 Q token 位置在 score
> 矩阵中占的 64 行**（详见 §1.4 的 S1G 视角）。本算子的调度以 S1G 行为最小行单位，
> 单独估算每行的成本。

#### 变化源 ①：Win 部分受窗口下限制约

Win 窗口理论最多 128 个 K token（`winLeft=127`），但**早期 Q 看不到那么多**。

以「第一次 prefill 一段长 5000 的序列」（S1 = S2 = 5000，无 KV 历史）为例：

| Q 行 | K 范围 (clip 后) | Win token 数 |
|------|------------------|--------------|
| 0    | [0, 0]           | **1**   ← 没历史，只能看自己 |
| 50   | [0, 50]          | **51**  ← 历史不够 128，被左 clip |
| 127  | [0, 127]         | **128** ← 刚好攒够 |
| 200  | [73, 200]        | 128     ← 窗口稳定滑动 |
| 4999 | [4872, 4999]     | 128 |

代码里 `s2FirstToken = i - 127`，算出负数会被 `Clip` 到 0，所以序列前 128
个位置的 Win 在增长，之后稳定 128。

#### 变化源 ②：Cmp 部分随历史长度增长，可能封顶

`cmp_kv` 是 `ori_kv` 的压缩版（按 `cmpRatio` 压缩）。Cmp 部分要看的 token 数 ≈
当前历史长度 / `cmpRatio`，所以越靠后的 Q 看的越多：

| 场景 | 公式 | 变化规律 |
|------|------|----------|
| **CFA**（无 topk） | `Cmp 长度 = 历史长度 / cmpRatio` | 一直随历史增长，不封顶 |
| **SCFA**（有 topk=512） | `Cmp 长度 = min(上式, 512)` | 涨到 512 后封顶 |

例（`cmpRatio=128, cmpTopK=512`, SCFA）：

```
   历史长度        Cmp token 数
   ───────────────────────────
   100             0      ← 还没启动 (100/128 取整 = 0)
   1000            7
   10000           78
   65536           512    ← 封顶
   100000          512    ← 保持封顶
```

#### 变化源 ③：尾块大小（按 512 切块凑不齐时）

**切的是什么？** —— 这一行（1 个 Q 位置）在 S2 方向上的 K token 序列。
而且 Win 和 Cmp **分开切**，因为它们的 K 数据来源不同：

- Win 部分用 `ori_kv` 里的原始 K
- Cmp 部分用 `cmp_kv` 里的压缩 K

两份 K 在内存里是独立 tensor，cube 一次矩阵乘只能吃一块连续的 K（B tile 必须连续），
所以**必须切成两组独立的基本块**。

**怎么切？** —— 每段独立按 S2 方向**每 512 个 K token 一刀**。
M 方向自动就是 64（1 个 Q 位置 × 64 个头，详见 §1.4），所以只切 S2 方向。
每个切片就是一个 `[64, 512]` 的基本块。

**最后一块往往不满 512**，叫尾块：

```
   Win 范围 = 128 个 token, 切 512 一块
     → 1 个块, 尾块大小 = 128 (剩 384 是空 padding)

   Cmp 范围 = 250 个 token
     → 1 个块, 尾块大小 = 250

   Cmp 范围 = 600 个 token
     → 2 个块: 1 个满块 (512) + 1 个尾块 (88)
```

尾块的实际算量通常小于满块。成本公式（详细解释见 §4.2）：

```
   cost = 6 × ⌈M/16⌉ + 10 × ⌈S2/64⌉
              ──────        ──────
        M 对齐到 16 后的份数  S2 对齐到 64 后的份数
   (6 和 10 是硬件实测的权重系数;
    16 和 64 是 cube 搬运颗粒)
```

S2 越小，`⌈S2/64⌉` 越小（注意是按 64 向上取整，所以是按档位跳变的，不是连续变化）。

所以**即使两个 Q 位置切出来的块数相同，行的总成本也可能不同**：

```
   Q 位置 A:  Win 128 + Cmp 400
     → Win 1 个块 (尾块 128) + Cmp 1 个块 (尾块 400)
     → 共 2 个块, 成本 = WinCost(64, 128) + CmpCost(64, 400)

   Q 位置 B:  Win 128 + Cmp 500
     → Win 1 个块 (尾块 128) + Cmp 1 个块 (尾块 500)
     → 共 2 个块, 成本 = WinCost(64, 128) + CmpCost(64, 500)
                                                ─────────────
                                       块数相同, 但 Cmp 尾块不同
                                       → 行 A 和 行 B 总成本不同
```

#### 三个加起来：每行成本可能差几倍

```
   按 Q 位置看一行的实际成本：
   行 0:     Win 1 token  + Cmp 0            成本 ≈ 极小
   行 127:   Win 128      + Cmp 0            成本   小
   行 5000:  Win 128      + Cmp 39 token     成本   中
   行 99999: Win 128      + Cmp 512 满       成本   最大
```

如果按 "总块数 / 24" 静态平均分核，前面分到轻活的核会闲死、后面分到重活的核会累死。
**所以本算子必须按行先估出真实开销，再做均衡分配。**

### 1.3 NPU 多核：为什么需要「调度」？

NPU 上有几十个 AI Core（本算子默认按 24 个 AIC + 48 个 AIV 来分），
它们可以**并行**干活。如果只让一个核干，速度就慢；让 24 个核一起干，
理论上能快 24 倍。

但问题来了 —— 如何把任务**均匀**地分给 24 个核？

```
   坏的分法 (不均衡)：
   核 0: ████████████████████████   ← 累死
   核 1: ███████                    ← 闲死
   核 2: ████                       ← 更闲
   ...
   总时间 = 最慢那个核的时间 = 24 单位

   好的分法 (均衡)：
   核 0: ███████████
   核 1: ███████████
   核 2: ███████████
   ...
   总时间 = 11 单位（快了一倍多！）
```

**本算子的全部价值就在于：算出第二种「好的分法」**，
让 24 个核的负载尽量平均。

### 1.4 为什么 Q 要把 S1 和头数合并：S1G 的来历

回到参数限制：`N1 = 64`，`N2 = 1`。

- N1 = 64 ：Q 有 **64 个注意力头**
- N2 = 1  ：K 和 V 只有 **1 个注意力头**

```
     Q 的结构 (N1=64 个头)            K / V 的结构 (N2=1 个头)
     ┌──────────────────────┐        ┌──────────────────────┐
     │  头 0:  ▓▓▓▓▓▓▓▓▓▓  │        │  头 0:  ░░░░░░░░░░  │
     │  头 1:  ▓▓▓▓▓▓▓▓▓▓  │        └──────────────────────┘
     │  ...                 │                ↑
     │  头 63: ▓▓▓▓▓▓▓▓▓▓  │       只有这一个 K/V 头
     └──────────────────────┘
```

这是 **Multi-Query Attention (MQA)** 架构：64 个 Q 头**共享同一份 K 和 V**。
`G = N1 / N2 = 64`，意思是「每 1 个 K/V 头要服务 64 个 Q 头」。

#### 不优化的做法：逐头算 64 次小矩阵乘

```
   for 头 h = 0..63:
       Q_h     = Q[:, h, :]              shape [S1, D]
       score_h = Q_h @ K.T               shape [S1, S2]
       ...
   K 要从 HBM 显存里读 64 遍 ← 带宽白白浪费
```

#### S1G 优化：把 64 个头沿 S1 维度展平，做一次大矩阵乘

```
   传统视角 (S1=4):                 S1G 视角 (S1G = 4×64 = 256):
   ┌─────────────────┐              ┌──────────────────┐
   │ pos0 ┬ h0   ▓D │              │ 行 0:  pos0, h0  │
   │      ├ h1   ▓D │              │ 行 1:  pos0, h1  │
   │      ├ ...     │              │ 行 ...            │
   │      └ h63  ▓D │              │ 行 63: pos0, h63 │
   │ pos1 ┬ h0   ▓D │ 重新排列 ►   │ 行 64: pos1, h0  │
   │      ├ ...     │              │ ...               │
   │      └ h63  ▓D │              │ 行 127:pos1, h63 │
   │ pos2 ┬ ...     │              │ ...               │
   └─────────────────┘              └──────────────────┘
   shape [S1, N1, D]                shape [S1G, D]
```

然后只做**一次**矩阵乘：

```
              K.T (D × S2)
              ┌──────────────┐
              │              │
              └──────────────┘
                     ×
   Q (S1G × D)              score (S1G × S2)
   ┌────────────┐           ┌────────────────┐
   │ pos0, h0  │            │                │
   │ pos0, h1  │            │  256 × 1500    │ ← 一次性算完所有头
   │ ...        │  ──────►  │                │
   │ pos3, h63 │            │                │
   └────────────┘           └────────────────┘
```

收益：
1. **K 只读 1 次**，带宽利用率 ↑
2. **矩阵变大** → NPU cube 算力利用率 ↑
3. 64 个头的结果在一次乘法里全出来了

#### 顺带解释了基本块尺寸

代码里 `mBaseSize_ = groupSize_ = 64`。这不是巧合 —— **每 64 行正好是一个 token
位置的全部 64 个头**：

```
   按 S1G 方向切块 (mBaseSize=64):
   ┌─────────────────────┐ ─┐
   │ pos0, h0           │  │
   │ pos0, h1           │  │ 块 0 = pos0 的 64 个头 (大小 64 × D)
   │ ...                │  │
   │ pos0, h63          │  │
   ├─────────────────────┤ ─┤
   │ pos1, h0 ~ h63     │  │ 块 1 = pos1 的 64 个头
   ├─────────────────────┤ ─┘
   │ ...                │
```

S2 方向每 512 列再切一刀，一个基本块就是 `[64, 512]` —— 至于为什么这两个数
是 NPU cube 单元的甜蜜点，详见 §2.1。

## 2. 关键概念定义

读后续内容前，先熟悉这几个名词：

| 名词 | 含义 |
|------|------|
| **B (Batch)** | 一次处理几条独立的样本（比如几句话） |
| **S1** | Q 这一边的序列长度（有多少 token） |
| **S2** | K 这一边的序列长度（有多少 token） |
| **N1 / N2** | Q / K 的「头数」（一种结构上的并行维度），本算子要求 N1=64, N2=1 |
| **G (groupSize)** | N1 / N2 = 64，表示 64 个 Q 头共享同一份 K（MQA 结构） |
| **S1G** | 把 64 个头沿 S1 展平后的维度（详见 §1.4），分块时按它来切 |
| **基本块 (basic block)** | 最小的计算单位，M 方向大小 = 64，S2 方向大小 = 512 |
| **AIC / AIV** | NPU 上的两种核心：AIC 算矩阵乘（cube），AIV 算向量（vector） |
| **FA** | FlashAttention，主计算阶段（AIC 做） |
| **FD** | FlashDecode，跨核归约阶段（AIV 做） |

### 2.1 为什么要切成「基本块」，以及为什么是 [64, 512]？

#### 第一步：score 矩阵 = Q 数 × K 数

Cube 单元做的是矩阵乘 `A @ B = C`。在 attention 里：

```
                  K^T : shape [D, S2]
                  ┌─────────────────────┐
                  │  (K 转置后)          │  ← D 行
                  │  每"列"对应 1 个 K   │
                  │   token 的向量       │
                  └─────────────────────┘
                          ×
   Q : shape [M, D]       score : shape [M, S2]
   ┌──────────┐           ┌─────────────────────┐
   │ M 行     │           │ M 行  ×  S2 列      │
   │ (每行=1  │  ───────► │                     │
   │  个 Q)   │           │ score[i,j] = Q_i·K_j│
   │ D 列     │           │                     │
   └──────────┘           └─────────────────────┘
```

关键事实：**score 矩阵的形状就是「Q 数 × K 数」**：
- **行 i** 对应**第 i 个 Q**（S1G 视角下是 "某 token 位置的某个头"）
- **列 j** 对应**第 j 个 K token**

所以下文说「M 方向切 64」和「64 个 Q」、「S2 方向切 512」和「512 个 K」
**是同一件事的两种说法**，不是两套概念。

#### 第二步：完整任务切成 [64, 512] 的方块

一个完整 attention 任务的 score 矩阵可能很大（比如 6400 × 1500）。
NPU 没法一口吞下，必须切成小方块：

```
       S2 方向（K 长度）= 1500 个 token
       ┌────────┬────────┬────────┐
       │ 块 0   │ 块 1   │ 块 2   │
       │ 512    │ 512    │ 476    │  ← 最后一块叫「尾块」
   ▲   │        │        │        │
   │   ├────────┼────────┼────────┤
   M   │        │        │        │
   方  │        │        │        │
   向  ├────────┼────────┼────────┤
   ↓   │        │        │        │
   =   │        │        │        │
   384 └────────┴────────┴────────┘
   (6  64 行一个块，整张图被切成 6×3 = 18 个块
   pos)
```

调度的「最小颗粒」就是这些 `[64, 512]` 的方块。本算子要决定：哪些块给哪个核。

#### 第三步：[64, 512] 这两个数从哪来？—— Cube 单元的甜蜜点

> 这一段稍微深入一点硬件，理解了它你就明白为什么 `mBaseSize_=64`、`s2BaseSize_=512`
> 是硬编码常量。可以先跳过，不影响理解后面的调度算法。

Cube 单元做的是矩阵乘**累加** (MMA)，但它的硬件颗粒**非常小**：每个 cycle 只算
类似 `[16,16] × [16,16] → [16,16]` 的小矩阵（FP16 输入累加到 FP32）。

`[64, 512]` 不是「一个 cycle 算的」，而是**软件层面切出来的 tile**，里面包含很多个
cycle 级别的 MMA。三个 tile 是**绑定的**：

```
   ┌─────────────────────────────────────────────────────┐
   │  A tile: [64, D]    ← Q 的一段 (M=64 行, D 列全)    │
   │  B tile: [D, 512]   ← K^T 的一段 (D 行全, N=512 列) │
   │  C tile: [64, 512]  ← 输出的一个 64×512 方块         │
   └─────────────────────────────────────────────────────┘
        ▲                            ▲
        │                            │
    M_tile=64 决定 A 的高         N_tile=512 决定 B 的宽
    同时是 C 的高                  同时是 C 的宽
```

所以「`[64, 512]` 是基本块的尺寸」严格说指的是 **C tile**（输出）的形状，
但 `M=64`、`N=512` 这两个数同时锁定了 A、B tile 的形状。三者一组，不能分开。

为什么是这两个数？—— 综合 4 个约束的折中：

```
   NPU 存储层级 (从上到下越快但越小):
   ┌─────────────────────────────────┐
   │  HBM  (主显存, GB 级, 最慢)      │
   └──────────────┬──────────────────┘
                  │ HBM ↔ L1
                  ▼
   ┌─────────────────────────────────┐
   │  L1 buffer (A3 上 512 KB)       │
   └──────────────┬──────────────────┘
                  │ L1 ↔ L0
                  ▼
   ┌─────────────────────────────────┐
   │  L0A / L0B / L0C (KB 级, 最快)   │  ← Cube 直接吃这一层
   └──────────────┬──────────────────┘
                  │ 每 cycle 喂一组 [16,16]×[16,16]
                  ▼
              Cube 单元 (MMA)
```

| 约束 | 太小会怎样 | 太大会怎样 |
|------|----------|----------|
| **Cube 算力利用率** | tile 太小 → MMA pipeline 填不满 → 算力空转 | （由其他约束限制） |
| **L0A 容量** | — | M=64 再大 → A tile `[M, D]` 装不下 L0A → 要再拆 |
| **L0C 容量 (C 累加器)** | — | M 或 N 再大 → C tile `[M, N]` 装不下 L0C |
| **A (Q) 复用 / L1↔L0 带宽** | N=512 再小 → 同一个 A 在更多基本块间反复加载 → 带宽浪费 | — |

> **关于 B 的搬运**：B tile `[D, 512]` 本来就超过 L0B (64KB)，**必须按 cube 内
> S2_chunk=64 分片加载**, 这跟基本块尺寸 N=512 无关 (跟硬件 cube 颗粒有关, 详见 §4.2)。
> **B 的总搬运量 = D × S2_total**, 跟 N_tile 大小无关 (N=512 还是 256, B 都得整个搬一遍)。
>
> **那 N=512 选大真正赚在哪?** 不是单一原因, 是 4 个维度的综合权衡:
>
> | 维度 | N 大 (如 512) | N 小 (如 256) |
> |------|---|---|
> | B 总搬运 | 一样 | 一样 |
> | **多核分核时 A 的总搬运** | 少 (任务分给更少核, 每核独立加载 A 一次) | **多** (任务分给更多核, 每核都独立加载自己的 A) |
> | **跨核行 / FD 归约任务** | 少 (一行块数少, 易整行给一个核) | 多 (一行被切碎, 更可能被多核瓜分) |
> | **Cube MMA pipeline 饱满度** | 高 (在 N 方向连续算更多 cube 指令) | 低 (基本块结束就要重启 pipeline) |
> | 并行度 (能同时算的核数) | 低 | 高 |
>
> 多核场景下 A 不能跨核复用 (每个 AIC 有独立 L0A), 所以"任务切得越碎 → 分给的核越多 → A 总搬运越大"。
> 同时 N 小还增加跨核行概率和 pipeline 重启次数。
> N=512 是**在 24 核已经够并行的前提下, 用并行度换 A 复用 + FD 减少 + pipeline 饱满**的工程权衡。
> 如果硬件有更多核, N 可能反而要选小 —— **这不是物理定律, 是工程权衡**。

所以 `[64, 512]` 实际是：
- **存储侧**：A tile 和 B 切片**精确填满** L0A 和 L0B (各 64 KB) ← **决定性约束** (见下面"关键事实")
- **算力侧**：让 cube 的 MMA pipeline 能基本填满
- **带宽侧**：B 切片正好等于 L0B 容量, 每次加载满载传输

代码里写死在：

```cpp
mBaseSize_  = groupSize_;   // = 64,  对应 M_tile
s2BaseSize_ = 512U;          //         对应 N_tile
```

**"吞吐"指的不是单一项**（搬运或算力），而是上面四个约束都不至于成为瓶颈时的综合最优。

> **关键事实：D=512 是固定值, 不是变量**
>
> 看似 `head_dim` 是 metadata 算子的属性输入, 但下游 `sparse_attn_sharedkv` 的
> tiling 里硬编码 `qHeadDim_ != 512` 报错:
>
> ```cpp
> constexpr uint32_t DIM_LIMIT = 512;
> OP_CHECK_IF(qHeadDim_ != DIM_LIMIT, ...)
> ```
>
> 所以**整套算子专门服务于 D=512** (对应 DeepSeek MLA 架构里 `kv_lora_rank=512` 的设计)。
>
> **这才是 `[64, 512]` 基本块尺寸的真正物理来源** —— 每一层 buffer 都被 D=512 精确反推填满:
>
> | 数据 | 大小 (D=512, FP16) | 对应 buffer |
> |------|---------------------|----------------|
> | A tile = `[64, 512]`        | 64  × 512 × 2 = **64 KB**  | **正好填满** L0A (64 KB)  |
> | B tile (基本块) = `[512, 512]` | 512 × 512 × 2 = **512 KB** | **正好填满** L1  (A3: 512 KB) |
> | B 切片 (cube 颗粒) = `[512, 64]`  | 512 × 64  × 2 = **64 KB**  | **正好填满** L0B (64 KB)  |
>
> 这不是"工程师拍脑袋调出来的甜蜜点", 而是**硬件 buffer 容量反推回去, 在 D=512 下精确
> 填满 L0A / L0B / L1 三层的唯一组合**。`M=64`、`s2BaseSize=512`、`cube 内 S2_chunk=64`
> 三个数字分别让 L0A、L1、L0B 一字不漏用满。
>
> 反过来理解: **基本块 N=512 不是更大就更好** —— 上限正是 L1 容量, `N = L1 / D / 2byte = 512KB / 512 / 2 = 512`。
> 想让 cube 算一个基本块时 B **只从 HBM 搬一次到 L1**, N 就只能取到 512。再大就装不下, 必须分次从 HBM 重搬, 反而退化。
>
> 如果模型换成 D=128 (LLaMA 类), 这套尺寸就完全不优了:
> - A tile `[64, 128]` × 2byte = 16 KB, 只用了 L0A 的 1/4
> - B 切片 `[128, 64]` × 2byte = 16 KB, 同样只用 1/4
> - 那时 M 可以放大到 256, S2_chunk 可以放大到 256, 才能填满 L0
>
> 所以 `[64, 512]` **不是普适的 "NPU 黄金尺寸", 是为 D=512 的 DSV4/MLA 配置精确设计的**。
> cost 公式里的 6 和 10 也是在 D=512 上 profile 拟合的, 换 D 后需要重新 profile。

#### 第四步：一个基本块到底算了什么？—— 不止 Q @ K^T

到这里我们一直拿 `Q @ K^T` 举例, 但**一个基本块的"工作"远不止这一步**。
完整 attention 三步在 FlashAttention 算法下是**融合执行**的:

```
   for each S2 block (= 一个基本块的 S2 范围):
       partial_score = Q @ K_block^T              ← cube 算 (产出 [64,512] score tile)
       partial_prob  = exp(partial_score - max)    ← vector 算 softmax
       partial_O    += partial_prob @ V_block      ← cube 算 (产出 [64,D] 输出贡献)
       online 更新 m, l
```

所以一个基本块对应**完整 attention 的整套**(矩阵乘 + softmax + 加权和), 不只是 `Q @ K^T`。

##### K 和 V 实际上是同一份数据 (这就是 "SharedKV" 的来历)

注意算子名 **Sparse Attn _SharedKV_ Metadata** 里的 "SharedKV" —— **K 和 V 共享同一份 tensor**:

- 下游算子输入里**没有 V**, 只有 `ori_kv` 和 `cmp_kv`, 同时充当 K 和 V
- README 公式明确写: $\tilde{K}=\tilde{V}$
- 这对应 DeepSeek **MLA (Multi-head Latent Attention)** 架构: KV 压缩成同一个 latent 向量
  (`kv_lora_rank=512`, 也就是 D=512 的由来)

```
   latent KV 矩阵 [S2, D=512]:    ← 一份 tensor, 双重身份
   ┌──────────────────┐
   │ block 0           │    算 Q @ K^T 时 → 当 K 用
   ├──────────────────┤    算 prob @ V 时 → 当 V 用
   │ block 1           │
   ├──────────────────┤
   │ ...               │
   └──────────────────┘
   按 S2 每 512 一刀
```

**一个基本块 ≡ 一段 latent KV + 完整 attention 计算**。

##### 额外收益：cube 算 prob @ V 时不需要再次搬运 V

因为 V 物理上就是 K, **从 HBM 搬一份 latent 就服务了两次矩阵乘** (`Q @ K^T` 和 `prob @ V`)。

- 传统 attention: K 搬一遍 + V 搬一遍 = `2 × D × S2_total`
- SharedKV (MLA): latent 搬一遍 = `D × S2_total`

这是 MLA 架构带来的 **2 倍带宽节省**, 也部分解释了为什么 cost 公式里的 6 和 10 这么"温和" — V 的访存基本被 K 的访存覆盖了。

##### cube 和 vector 协作是硬件绑定, metadata 不管

昇腾架构 **AIC : AIV = 1 : 2 比例固定**, 每个 AIC 自动配两个 AIV:

```
   AIC 0 ─┬─ AIV 0  ┐
          └─ AIV 1  ┴── 这一组协作完成一个基本块的全部计算

   AIC 1 ─┬─ AIV 2  ┐
          └─ AIV 3  ┴── 另一组
```

AIC 算完 `Q @ K^T` 把 score tile 通过共享 UB 传给配对的 AIV, AIV 算 softmax 再传回。
**这种协作完全由硬件流水线自动完成, metadata 不介入**。

##### metadata 只显式分配两件事

```
   FA 段 (AIC): 哪个 AIC 算哪些基本块 (配对的 AIV 自动跟着干)
   FD 段 (AIV): 跨核归约 — 当一行被多个 AIC 瓜分时, 各 AIC 算出部分 (O', m, l)
                写到 workspace, 这时需要"别的" AIV 来 rescale 合并 (详见 §2.2)
```

FD 段才是"额外的" AIV 任务, 因为它要合并跨核结果。
基本块**内部**的 cube-vector 协作是硬件流水线, 不需要 metadata 分配。

##### cost 公式也覆盖了完整 attention

cost 公式 `6 × ⌈M/16⌉ + 10 × ⌈S2/64⌉` 中, 6 和 10 是 profile 实测拟合的,
**已经包含 cube 算 Q@K^T + vector 算 softmax + cube 算 prob@V 的总耗时**, 不只是 Q@K^T 这一项。

而且两次矩阵乘的计算量近似相等:
- `Q @ K^T`: 计算量 ≈ M × S2 × D
- `prob @ V`: 计算量 ≈ M × S2 × D

所以用 M 和 S2 这两个维度就能涵盖整个基本块的成本估算。

### 2.2 切完之后怎么合并？—— online softmax

> 这一节解释一个非常容易被忽略的细节：**S2 方向切了 512 后，几个块的结果不是简单地逐元素加**。
> 理解这一点是理解后面 FD 阶段的前提。

完整 Attention 是：

```
   score = Q @ [K_0 | K_1]^T            shape [S1, 1024]
   prob  = softmax(score, axis = S2)    ← 沿整个 S2 归一化
   O     = prob @ [V_0 | V_1]
```

如果只算 block 0 (S2 的前 512 列)：

```
   score_0 = Q @ K_0^T                  shape [S1, 512]
   prob_0  = softmax(score_0)           ← 局部归一化！

   局部 prob_0 的分母  = Σ exp(score_0)
   正确  prob   的分母  = Σ exp(score_0) + Σ exp(score_1)
                                          ─────────────
                                          block 0 算的时候不知道这部分
```

所以**直接把局部结果加起来 = 错的**。

#### Flash Attention 的解法：每块多保留两个统计量

每个块算完不只输出 O'，还要带上两个「账本」。注意 softmax 是沿 S2 方向 (行内)
归一化的, 所以这两个账本是**每行一个值的向量**, 不是标量:

```
   每块输出 3 件东西:
     O' = prob_local @ V_block       ← 部分输出, shape [S1, D] = [64, 512]
     m  = max(score_block, axis=S2)   ← 每行的最大值, shape [S1] = [64]
     l  = Σ exp(score_block - m, axis=S2)  ← 每行的归一化分母, shape [S1] = [64]
```

也就是说, `m` 和 `l` 都是**长度 = S1 = 64 的向量** —— 每个元素对应基本块里的一行 Q,
是该行在 S2 方向 reduce 之后的结果:

```
              S2 方向 (K token) →
              ┌─────────────────────┐    ┌────┐
   ▲          │  ←─── max along S2 ─→│ →  │m[0]│
   │          ├─────────────────────┤    ├────┤
   S1=64      │  ←─── max along S2 ─→│ →  │m[1]│
   行          │       ...            │    │... │
   ↓          ├─────────────────────┤    ├────┤
              │  ←─── max along S2 ─→│ →  │m[63]│
              └─────────────────────┘    └────┘
              [64, 512] score_block       [64] m
```

这些**逐行 reduce** 操作在硬件上由 **AIV (vector)** 完成 — cube 算完 score_block 后
通过 UB 传给配对 AIV, AIV 算出 `m`, `l`, `prob` 再传回 cube 做 `prob @ V`。

合并两块用「rescale + 加权和」公式：

```
   m_new = max(m_0, m_1)

   l_new = l_0·exp(m_0 - m_new) + l_1·exp(m_1 - m_new)
           ─────────────────────  ─────────────────────
           把两块的分母拉到同一基准再相加

   O_new = ( O'_0 · l_0 · exp(m_0 - m_new)
           + O'_1 · l_1 · exp(m_1 - m_new) ) / l_new
```

##### 公式从哪来？—— 一步步推导

> 想知道这个公式不是从天上掉下来的, 看这一节。觉得"知道用就行"可以跳过。

**起点**: 全局 softmax 定义 (对一行 Q 来说; 多行独立处理):

```
   m   = max(全 score)                          ← 全局最大值 (稳定化常数), 标量
   l   = Σ_j exp(score[j] - m)                  ← 全局归一化分母, 标量
   O   = ( Σ_j exp(score[j] - m) · V[j] ) / l   ← 全局输出, 长度 D 的向量
```

> **关于 O 的求和**: `V[j]` 是 V 矩阵第 j 行 (**长度 D 的向量**), `exp(score[j]-m)/l`
> 是标量 (softmax 权重)。"标量 × 向量" 得到向量, "向量 + 向量" 是逐元素相加。
> 所以 `Σ_j` 把 S2 个长度 D 的向量按位置加起来, 结果是长度 D 的向量。
>
> 分量展开: `O[k] = Σ_j w_j · V[j][k]` —— **O 的第 k 维 = V 矩阵第 k 列的加权和**, D 维各自独立做一次。
> 这等价于矩阵乘 `prob @ V` (行向量 [1, S2] × 矩阵 [S2, D] = 行向量 [1, D])。
>
> 这就是 attention 的"对 V 加权求和"本质 —— **O 是 V 各行的加权平均**, 权重由 Q · K^T 经 softmax 得到。

**问题**: 切块时, 块 0 不知道块 1 的 score, 只能用**局部** max 算指数:

```
   m_0  = max(score_0)
   l_0  = Σ_{j∈块0} exp(score_0[j] - m_0)
   O'_0 = (1/l_0) · Σ_{j∈块0} exp(score_0[j] - m_0) · V_0[j]   ← 局部归一化
```

**关键代数恒等式** (rescale 因子的来源):

```
   exp(score_0[j] - m)
   = exp((score_0[j] - m_0) + (m_0 - m))
   = exp(score_0[j] - m_0) · exp(m_0 - m)
                              ─────────────
                  跟 j 无关, 可以提取出求和
```

**第 1 步: 把块 0 对"全局分子"的贡献用 O'_0 表达**

```
   Σ_{j∈块0} exp(score_0[j] - m) · V_0[j]

   = exp(m_0 - m) · Σ_{j∈块0} exp(score_0[j] - m_0) · V_0[j]   ← 提取因子
                    ────────────────────────────────────────
                                = l_0 · O'_0  (由 O'_0 定义)

   = exp(m_0 - m) · l_0 · O'_0
```

块 1 同理 → `exp(m_1 - m) · l_1 · O'_1`。

**第 2 步: 全局 l 也用同样手法拆** (跟第 1 步是同一个"提因子"技巧)

① **按块拆求和** (纯粹分段, 没用变换):

```
   l = Σ_j exp(score[j] - m)
     = Σ_{j∈块0} exp(score_0[j] - m)  +  Σ_{j∈块1} exp(score_1[j] - m)
```

② **处理块 0 那项**, 用关键恒等式 + 提因子:

```
   Σ_{j∈块0} exp(score_0[j] - m)

   = Σ_{j∈块0} [exp(score_0[j] - m_0) · exp(m_0 - m)]   ← 代入恒等式

   = exp(m_0 - m) · Σ_{j∈块0} exp(score_0[j] - m_0)     ← 因子跟 j 无关, 提出来
                    ───────────────────────────────
                    = l_0 (这就是 l_0 的定义)

   = exp(m_0 - m) · l_0
```

③ **块 1 完全同理**:

```
   Σ_{j∈块1} exp(score_1[j] - m) = exp(m_1 - m) · l_1
```

④ **合起来**:

```
   l = exp(m_0 - m) · l_0  +  exp(m_1 - m) · l_1
```

直觉: 把每块的局部分母 `l_i` 乘 `exp(m_i - m)` "拉"到全局基准上, 再相加。
跟第 1 步"拉 O 的分子"是同一个动作, 只是作用对象换成了分母 `l`。

**第 3 步: m 取多少?** 代数上 m 可任取, 但为了数值稳定 (避免 exp 溢出) 取:

```
   m_new = max(m_0, m_1)   ← 也正好等于全局 max, 因为 "max 的 max = 总 max"
```

**合起来**:

```
   l_new = exp(m_0 - m_new) · l_0 + exp(m_1 - m_new) · l_1
   O_new = ( exp(m_0 - m_new) · l_0 · O'_0
           + exp(m_1 - m_new) · l_1 · O'_1 ) / l_new
```

这就是上面那个公式。

**直觉总结**: 三步操作

1. **撤销局部归一化**: 把 `O'_i` 乘回 `l_i` —— 还原成"未归一化的原始累加和"
2. **统一指数基准**: 乘 `exp(m_i - m_new)` —— 把基准从 `m_i` 改成 `m_new`
3. **加起来再除以新分母**: 像普通加权求和一样合并

**N 块推广** (FD 实际场景: 一行可能被 N 个核瓜分, N 通常 > 2)

代码里 `fdS2SplitNum`、`maxS2SplitNum` 没有 2 的上限 —— 一个 S1G 行
如果很重, 被 3 个、4 个甚至更多核瓜分都可能, 所以 FD 实际要合并的部分结果数可以是 N。
N 块情形的公式跟 2 块同一推导, 把求和范围扩到 N:

```
   m_new = max(m_0, m_1, ..., m_{N-1})

   l_new = Σ_{i=0}^{N-1}  l_i · exp(m_i - m_new)

   O_new = ( Σ_{i=0}^{N-1}  O'_i · l_i · exp(m_i - m_new) ) / l_new
```

实现可以选两种**数学等价**的形式 (用归纳法可证):

**形式 A — N 路并行合并** (一步到位):

```
   ① m_new = max over i
   ② α_i   = exp(m_i - m_new)     (i = 0..N-1)
   ③ l_new = Σ α_i · l_i
   ④ O_new = (Σ α_i · l_i · O'_i) / l_new
```

**形式 B — 增量合并** (两两合并 N-1 次):

```
   m_acc = m_0, l_acc = l_0, O_acc = O'_0
   for k = 1, 2, ..., N-1:
       m_new = max(m_acc, m_k)
       α     = exp(m_acc - m_new)
       β     = exp(m_k   - m_new)
       l_new = α · l_acc + β · l_k
       O_new = (α · l_acc · O_acc + β · l_k · O_k) / l_new
       (m_acc, l_acc, O_acc) = (m_new, l_new, O_new)
```

**两种形式用在不同场景**:

| 场景 | 用哪种 | 原因 |
|------|--------|------|
| 核内 FlashAttention 主流程 | **增量** (B) | K block 一次一块流式送进来, 必须边算边累计 |
| 跨核 FD 阶段 | **A 或 B 都行** | N 个核的部分结果已经全部写在 workspace 里, AIV 一次性 N 路也合法 |

图示：

```
   block 0 算完:        block 1 算完:
   ┌──────────┐         ┌──────────┐
   │ O'_0     │         │ O'_1     │
   │ m_0      │         │ m_1      │
   │ l_0      │         │ l_1      │
   └──────┬───┘         └────┬─────┘
          \                  /
           \   rescale +    /
            ▼  加权求和   ▼
          ┌─────────────────┐
          │  最终的 O       │
          └─────────────────┘
```

#### 合并在哪里发生？—— 两种情况

```
   情况 A：一行的所有 S2 块都被分给同一个核
   核 i:  ░│░│█│█│█    ← 自己边算边合 (online softmax 在核内完成)
   不需要跨核归约，metadata 里不出现 FD 任务

   情况 B：一行的 S2 块被多个核瓜分（「跨核行」）
   核 i-1 算:  ░│░│░│░         (S2 块 0~3)
   核 i   算:        █│█│█      (S2 块 4~6)
                ↓
   两个核各自写出 (O', m, l) 到 workspace
                ↓
   AIV 核 (FD 阶段) 执行 rescale 公式，合并出最终 O
```

**本算子里的 FD 阶段做的就是情况 B 的合并**。`RecordFDInfo`、`fdMetadata`、
`firstFdDataWorkspaceIdx` 这些字段，全是为了告诉 AIV：
「该去 workspace 哪里取 (O', m, l)、按上面公式合并、结果写到哪」。

**FD 不是逐元素加，它执行的是 rescale 公式**。

## 3. 输入与输出

### 3.1 输入：每个 batch 的实际长度

最常见的场景下，输入是这两个数组：

```
seqused_q  = [120,  80,  300, 200]    ← batch_size=4, 这 4 个样本 Q 各自的真实长度
seqused_kv = [1500, 100, 2000, 250]   ← batch_size=4, 这 4 个样本 K 各自的真实长度
                                        (K 比 Q 长很多, 因为推理时 K cache 包含历史)
```

> **术语澄清**: "batch" 是「一批样本」, "batch_size" 是「一批包含几个样本」。
> NLP 场景下每个样本通常 = 一句话/一段文本, 所以 `batch_size=4` 意味着这一批里有 4 句话。
> (下文用 "样本" 指代每一句, 用 "batch" 指代整批。)

**为什么需要「实际长度」？** 因为**同一批里的几个样本长度不一样** (比如这 4 个样本分别是
120, 80, 300, 200 个 token), 但 NPU 上的 tensor 必须是规整矩阵, 每行长度得一样:

```
   Q tensor: shape [batch_size=4, max_seqlen_q=300, ...]
   ┌─────────────────────────────────────────────────────────────┐
   │ 样本 0: [t0,t1,...,t119,  0,0,...,0]  ← 120 真实 + 180 padding │
   │ 样本 1: [t0,t1,...,t79,   0,0,...,0]  ← 80  真实 + 220 padding │
   │ 样本 2: [t0,t1,...,t299]               ← 300 真实 (满)         │
   │ 样本 3: [t0,t1,...,t199,  0,0,...,0]  ← 200 真实 + 100 padding │
   └─────────────────────────────────────────────────────────────┘
                                                                 ↑
                                  seqused_q = [120, 80, 300, 200]
                                  告诉算子每个样本真实部分到哪
```

所以短样本会被**补 0** 到批内最长长度 (300)。但**真正要算的只有 padding 之前那部分**,
padding 出来的 0 不该参与计算 (会污染 softmax 概率)。本算子需要 `seqused_q/kv` 来:
1. 准确估算每个样本的真实工作量 (按真实长度算成本, 不被 padding 误导)
2. 让下游算子知道每个样本算到哪里就停

### 3.2 输出：metadata（一张 1024 个 int32 的表）

输出固定 1024 个 int32，里面其实分两段：

```
   metadata 内存布局 (1024 个 int32)
   ┌─────────────────────────────────────────────────┐
   │ FA 段: 36 个核 × 8 个字段 = 288 个 int32         │ ← AIC 用
   ├─────────────────────────────────────────────────┤
   │ FD 段: 72 个核 × 8 个字段 = 576 个 int32         │ ← AIV 用
   ├─────────────────────────────────────────────────┤
   │ 剩余空间（未使用，做对齐）                       │
   └─────────────────────────────────────────────────┘
```

每个 AIC 核拿到的「工作通知单」长这样（8 个字段）：

```
   faMetadata[核 i] = {
     [0] CORE_ENABLE          ← 1=要干活, 0=休息
     [1] BN2_START            ← 从哪个 (batch, kv_head) 起
     [2] M_START              ←   在这个 (batch, kv_head) 下, 从哪个 S1G 行起
     [3] S2_START             ←     在这个 S1G 行下, 从哪个 S2 块起
     [4] BN2_END              ← 干到哪个 (batch, kv_head) 止
     [5] M_END                ←   干到哪个 S1G 行止
     [6] S2_END               ←     干到该行的哪个块止
     [7] FIRST_FD_WORKSPACE   ← 跨核归约时, 结果存哪个槽位
   }
```

这就是工头给每个工人发的「任务卡」, 用三层嵌套维度 `BN2 → S1G → S2` 定位任务起止。

> **关于 BN2 和 S1G 里的 "head"**: 这两个 head 不是同一个东西!
>
> - **BN2** 里的 "N2" = **KV head 数** (K/V 的头数)
> - **S1G** 里的 "G" = **N1 / N2** = 每个 KV head 服务的 **Q head 数**
>
> 关系: 一个 KV head 给 N1/N2 = G 个 Q heads 共享 K/V。S1G 把这 G 个 Q heads
> 沿 S1 维度展平 (详见 §1.4), 所以 S1G 维度**已经包含 Q heads**。BN2 维度只是
> "哪个样本 × 哪个 KV head"。
>
> 本算子硬约束 **N2=1**, 所以:
> - `BN2_START / END` 实际就是 **batch 起止编号** (head 那一维只有 1 种)
> - G = N1/1 = 64, S1G = S1 × 64
>
> 字段叫 BN2 而不是 batch, 是按通用语义命名 — 让 N2 > 1 时可以直接用。

## 4. 算法主流程

整个调度算法分 5 步（对应 `BalanceSchedule` + `GenMetaData`）：

```
   ┌─────────────────────────────────────────────────┐
   │ Step 1: CalcSplitInfo                           │
   │  为每个 batch 算出切多少基本块                  │
   ├─────────────────────────────────────────────────┤
   │ Step 2: CalcCostInfo                            │
   │  估算每行 / 每 batch 的「开销」（成本分数）     │
   ├─────────────────────────────────────────────────┤
   │ Step 3: CalcSplitPlan                           │
   │  按平均开销，从前往后把块发给 24 个核           │
   │  （三级分配：按 batch → 按行 → 按块）          │
   ├─────────────────────────────────────────────────┤
   │ Step 4: SplitFD                                 │
   │  如果有「跨核行」需要归约结果，安排 AIV 来合并  │
   ├─────────────────────────────────────────────────┤
   │ Step 5: GenMetaData                             │
   │  把上面的分配方案写进 metadata 这张表           │
   └─────────────────────────────────────────────────┘
```

下面逐步详解。

### 4.1 Step 1: 切基本块

对每个样本, 根据它的 S1 (Q 长度) 和 S2 (K 长度) 算出在两个方向上各能切多少基本块。
用 §3.1 那 4 个样本的数据展开:

```
   样本 0:  S1 = 120,  S2 = 1500
            S1G = 120 × 64 = 7680
            M  方向块数 = ceil(7680 / 64)  = 120
            S2 方向块数 = ceil(1500 / 512) = 3    ← 2 个满块 + 1 个尾块
            尾块大小:  M tail = 0,  S2 tail = 1500 % 512 = 476

   样本 1:  S1 = 80,   S2 = 100
            S1G = 80 × 64 = 5120
            M  方向块数 = ceil(5120 / 64)  = 80
            S2 方向块数 = ceil(100 / 512)  = 1    ← 整个就是个 100 大小的尾块
            尾块大小:  M tail = 0,  S2 tail = 100

   样本 2:  S1 = 300,  S2 = 2000
            S1G = 300 × 64 = 19200
            M  方向块数 = ceil(19200 / 64) = 300
            S2 方向块数 = ceil(2000 / 512) = 4    ← 3 个满块 + 1 个尾块
            尾块大小:  M tail = 0,  S2 tail = 2000 % 512 = 464

   样本 3:  S1 = 200,  S2 = 250
            S1G = 200 × 64 = 12800
            M  方向块数 = ceil(12800 / 64) = 200
            S2 方向块数 = ceil(250 / 512)  = 1    ← 整个就是个 250 大小的尾块
            尾块大小:  M tail = 0,  S2 tail = 250
```

注意:
- M 方向 tail 都是 0, 因为 S1 × 64 都是 64 的整数倍 (G=64 整除)
- S2 方向 tail 取决于 S2 mod 512: 样本 0、2 切出 "满块 + 尾块" 混合, 样本 1、3 只有一个不到 512 的尾块
- 如果某个样本 S1=0 或 S2=0 (空样本), 这一步会输出 0 块, 后面调度直接跳过

**这一步的运算就两个"向上取整除法" (ceil)**:

```
   M  方向块数 = ceil(S1 × G / mBaseSize) = ceil(S1 × 64 / 64) = S1
   S2 方向块数 = ceil(S2 / s2BaseSize)    = ceil(S2 / 512)
```

各符号:

| 符号 | 取值 | 含义 (在哪解释过) |
|------|------|------------------|
| `S1` / `S2` | 各样本的真实长度 | 从 `seqused_q` / `seqused_kv` 取 (§3.1) |
| `G` | 64 | 每个 KV head 服务的 Q head 数 = N1/N2 (§1.4) |
| `mBaseSize` | 64 | 基本块的 M 边 (§2.1) |
| `s2BaseSize` | 512 | 基本块的 S2 边 (§2.1) |
| `ceil(a/b)` | 向上取整 | 例: `ceil(1500/512) = 3`, `ceil(100/512) = 1` |

**注意**: 因为 `G = mBaseSize = 64`, 上面 M 方向公式化简后 = `S1` —— 也就是
**每个 Q token 位置对应 1 个基本块** (一个 token 位置含 G=64 个 Q heads 沿 S1 展平,
正好填满一个 M=64 的基本块行)。

> **代码对照** (想看 C++ 源码可以看, 不影响理解):
>
> 这两行就在 `CalcSplitInfo` 函数里, ceil 用整数除法 `(a + b - 1) / b` 实现:
>
> ```cpp
> splitInfo.s1GBaseNum[bIdx] = (s1Size * groupSize_ + (mBaseSize_-1)) / mBaseSize_;
> splitInfo.s2BaseNum[bIdx]  = (s2Size + s2BaseSize_ - 1) / s2BaseSize_;
> ```
>
> 变量对应: `s1Size = S1`, `s2Size = S2`, `groupSize_ = G = 64`,
> `mBaseSize_ = 64`, `s2BaseSize_ = 512`, `bIdx` = 样本索引 (0 ~ B-1),
> `splitInfo.s1GBaseNum` / `s2BaseNum` = 算出来的 M / S2 方向块数。

### 4.2 Step 2: 估算开销

「开销」就是一个块大概要花多少时间。本算子用一个简单的公式估算：

```
   cost(block) = 6 × M_对齐块数 + 10 × S2_对齐块数
```

直觉是：M 方向每 16 行有 6 个单位的开销，S2 方向每 64 列有 10 个单位的开销。
这些系数是工程师在硬件上实测后定的「经验值」。

> **常见疑惑**：S2 方向每 64 列的数据量 = M 方向每 16 行的 4 倍 (都乘 D)，
> 按数据量比应该是 6:24, 为什么实际是 6:10?
>
> 因为公式是**加法**而不是乘法 —— 它捕捉的是「重启循环 / 加载新数据的边际开销」，
> 不是数据量本身。实际硬件上:
>
> - **A tile 在 L0A 可复用**: 一次加载后, M 方向遍历时不用重搬, 整个 S2 方向遍历都摊销同一份 A
> - **Cube pipeline 重叠**: B 的下一片搬运可以和 cube 当前计算并行, 掩盖大部分搬运耗时
> - **Burst 传输 + 预取**: 连续搬运的单位带宽利用率远高于理论值
>
> 所以 S2 方向的「等效边际开销」实测只有 10, 而不是理论数据量比对应的 24。
>
> 另外, 这个 cost 只用来**相对比较** —— 让每个核 `sum(cost)` 大致相等。
> 绝对值不重要, 相对比例对就够。6 和 10 是 profile 数据上拟合的「相对负载权重」,
> 不是精确的耗时模型。

> **继续追问**：为什么 M 颗粒是 16, S2 颗粒是 64? L0B 比 L0A 大吗?
>
> 不是 L0B 更大 —— Da Vinci 架构上 L0A 和 L0B **容量相同** (各 64KB)。
> 16 和 64 是 cube 内部**嵌套循环的步长**, 不是 tile 大小:
>
> ```
> for s2 in 0..512 step 64:       ← S2 方向每 64 列一组
>     for m in 0..64  step 16:     ← M 方向每 16 行一组
>         for k in 0..D   step 16:
>             cube_MMA([16,16] × [16,16])    ← 硬件 MMA
> ```
>
> - **M 颗粒 16**: 硬件 MMA 一次最少处理 `[16,16]`, M 方向天然按 16
> - **S2 颗粒 64**: **B tile 太大装不下 L0B**, 必须按 S2 切片. 本算子 D=512:
>     - A tile `[64, 512]` × 2byte = **64 KB**, **正好填满** L0A
>     - B tile `[512, 512]` × 2byte = **512 KB**, **远超** 64 KB 的 L0B
>     - 所以 B 必须切成 `[512, 64]` × 2byte = **64 KB** 一片片加载, **正好填满** L0B
>     - **S2 颗粒 64 不是巧合, 是为了让 B 切片正好等于 L0B 容量** (D=512 反推得 64)
>
> 这同时也是 cost 公式里 S2 系数 (10) 比 M 系数 (6) 大的根本原因:
>
> - 每多算 1 个 **M 颗粒** (16 行 M): B 加载量**不变** (B 切片跟 M 维度无关)
> - 每多算 1 个 **S2 颗粒** (64 列 S2): B 必须**从 L1 重新搬一片** [D, 64]
>
> 也就是 **S2 颗粒触发 B 加载, M 颗粒不触发** —— A 在 S2 方向遍历时一直留在 L0A 复用,
> 而 B 在 S2 方向每 64 列就要换一片。所以单位 S2 颗粒的成本就高过单位 M 颗粒。

#### 难点：Win 和 Cmp 范围每一行都不一样

注意之前讲的 mask 形状：

```
   行 0:  ░░░░  █                     ← Cmp 1 块, Win 1 块
   行 1:  ░░░░░ ██                    ← Cmp 1 块, Win 1 块
   行 2:  ░░░░░░ ██                   ← Cmp 1 块, Win 1 块
   行 N:  ░░░░░░░░░░░░ ██████          ← Cmp 多块, Win 多块
```

每行的「该看哪些 K 块」要单独算 —— 这正是 `CalcS1GCache` 干的：

1. **`CalcS2TokenRange`**：根据 mask 模式（band / right-down-causal）算出这行的 Q
   能看到 K 的哪个 token 范围 `[s2_first, s2_last]`。
2. **`CalcBlockRangeAndTailSize`**：把 token 范围转换成块编号，并算 Cmp 部分的范围
   （Cmp 部分一般固定 = `cmpTopK / 512` 个块）。
3. **`CalcCostTable`**：算这一行 4 种块组合（NORMAL × NORMAL、NORMAL × TAIL、
   TAIL × NORMAL、TAIL × TAIL）各自的成本。
4. **`CalcWinS1GCache` / `CalcCmpS1GCache`**：拼出 Win 部分和 Cmp 部分各自的总成本。
5. **`GatherWinAndCmpCache`**：汇总成这一整行的成本 `s1GCost` 和块数 `s1GBlock`。

最后所有行的成本相加，得到 `totalCost` —— 这是「全部工作量的总和」。

```
   理想平均开销 = totalCost / 24 (核数)
   = 这就是每个核「应该」分到多少开销
```

### 4.3 Step 3: 三级分配（核心难点）

调度的关键：怎么把一连串的「工作块」连续地切给 24 个核？
代码用了**三级分配**策略，从粗到细：

```
   想象工作量是一条长丝带：
   ┌─────────────────────────────────────────────────────────────┐
   │ Batch0 │ Batch1│ Batch2          │ Batch3       │ Batch4    │
   └─────────────────────────────────────────────────────────────┘
   要把它切成 24 段，每段长度尽量相等

   第一级 (AssignByBatch): 尽量按整 batch 切
   ──┬─────┬─────┬──────────┬──────────┬─────────┬───
     ↑     ↑     ↑           ↑          ↑
   核0/1   核2   核3         核4         核5
   如果一个 batch 太大放不下，进入第二级

   第二级 (AssignByRow): 把这个大 batch 按整行切
   ─── Batch3 ───────────
   行1│行2│行3│行4│行5│行6
       ↑        ↑
       核5      核6
   如果一行还是太大放不下，进入第三级

   第三级 (AssignByBlock): 把这一行按块切
   ── 行5 ──
   ░│░│█│█│█│█
   ↑   ↑
   核6 核7
```

为什么要分三级？—— 是为了让**同一个核处理的数据尽量连续**，
连续了内存访问就快，碎片化分配反而会变慢。

**容差机制**：每一级分配时不会严格卡 `costLimit`，而是允许一点点超出
（`IsWithinTolerance` 函数）。这避免了「差一点点就装下了，结果硬塞给下一个核」
的尴尬，让最终分配更均衡。

#### 跨核行的处理

如果第三级把**一行的中间**切给了两个核：

```
   行 X:
   核 i-1 干:  ░│░│░│░         (S2: 块 0 ~ 3)
   核 i   干:        █│█│█      (S2: 块 4 ~ 6)

   两个核都只算了行 X 的一部分结果，最终需要把两部分「合并」（归约）
   合并工作交给 AIV 核去做，这就是 FD 阶段
```

`RecordFDInfo` 会记录所有这种「跨核行」，存进 `fdRes` 结构。

### 4.4 Step 4: FD 阶段调度

> 前置概念：FD 做的「归约」**不是逐元素加**，而是 Flash Attention 的
> rescale 公式。如果还不清楚为什么需要这种归约，回看 §2.2。

`SplitFD` 函数把所有需要归约的行**再次分配**给 48 个 AIV 核：

```
   归约任务 0: 行 X，需要合并 2 份结果，M 方向 64 行
   归约任务 1: 行 Y，需要合并 3 份结果，M 方向 64 行
   归约任务 2: 行 Z，需要合并 2 份结果，M 方向 32 行
   ...

   按总负载 / AIV 核数 算出每个 AIV 该处理多少
   再为每个 AIV 分配：哪个归约任务的第几行起，处理几行
```

注意 FD 分配粒度更细 —— 同一个归约任务可以拆给多个 AIV。

### 4.5 Step 5: 写 metadata

最后把上面所有分配结果写进输出张量。`GenMetaData` 的逻辑非常直接：

```cpp
for 每个 AIC 核 i:
    if i 没被分到工作:
        faMetadata[i][CORE_ENABLE] = 0       // 这个核休息
    else:
        faMetadata[i][CORE_ENABLE]  = 1
        faMetadata[i][BN2_START]    = 上一个核的 BN2_END
        faMetadata[i][M_START]      = 上一个核的 M_END
        faMetadata[i][S2_START]     = 上一个核的 S2_END
        faMetadata[i][BN2_END]      = splitRes.bN2End[i]
        faMetadata[i][M_END]        = splitRes.gS1End[i]
        faMetadata[i][S2_END]       = splitRes.s2End[i]
        faMetadata[i][FD_WORKSPACE] = ...
```

FD 段同理。这一步纯粹是格式转换，没有算法逻辑。

## 5. 完整示例：把所有概念串起来

> **注**: 这一节用一个**独立的、更小的例子** (B=2, 不是 §3.1 / §4.1 用的 B=4),
> 目的是让端到端的三级分配过程便于跟踪。数字不需要跟前面对应。

假设：
- B = 2（一批 2 个样本）
- N1 = 64, N2 = 1（所以 G = 64）
- 样本 0: S1 = 100, S2 = 1000
- 样本 1: S1 = 50,  S2 = 600

**Step 1 切块**：

```
   样本 0:  S1G = 100×64 = 6400
             M  方向 = 6400 / 64 = 100 块
             S2 方向 = ceil(1000 / 512) = 2 块  (tail = 488)

   样本 1:  S1G = 50×64 = 3200
             M  方向 = 50 块
             S2 方向 = ceil(600 / 512) = 2 块   (tail = 88)
```

**Step 2 估开销**（简化，假设每行约 5 个 Win 块 + 1 个 Cmp 块 = 6 块）：

```
   样本 0:  100 行 × 6 块/行 ≈ 600 个块 ≈ 4800 单位开销
   样本 1:   50 行 × 6 块/行 ≈ 300 个块 ≈ 2400 单位开销

   totalCost ≈ 7200
   每核理想开销 ≈ 7200 / 24 ≈ 300
```

**Step 3 分配**（示意，不严格还原算法）：

```
   核 0:  样本 0 的行 0-1   (≈ 12 块, 300 单位)  ← 第二级 AssignByRow
   核 1:  样本 0 的行 2-3
   核 2:  样本 0 的行 4-5
   ...
   核 16: 样本 0 的行 32-33
   核 17: 样本 1 的行 0-5    ← 切到 样本 1
   ...
```

**Step 4 FD**：如果某行被两核共享，记录归约任务。

**Step 5 写出**：metadata 表里 24 个 AIC 核各自有起止位置。

## 6. 后续：metadata 怎么被使用？

下游算子 `SparseAttnSharedkv` 拿到 metadata 后：

```
   AIC 核 i 启动时：
   1) 读自己的 faMetadata[i][CORE_ENABLE]
   2) 如果是 1，按 BN2_START~END / M_START~END / S2_START~END
      去全局内存里把对应的 Q / K / V 数据取过来
   3) 算自己负责的那一段 Attention
   4) 写结果到 workspace 或者最终输出

   AIV 核 j 启动时：
   1) 读自己的 fdMetadata[j][CORE_ENABLE]
   2) 如果是 1，从 workspace 里拿到要归约的几份部分结果
   3) 算 softmax 的归约修正，写到最终输出
```

整张 metadata 表就是这两个阶段共用的「任务清单」。

## 8. 设计上的几个值得注意的点

### 8.1 为什么放在 AICPU 而不是 AICore？

AICore 擅长矩阵和向量计算，分核调度这种「带分支、带循环、不规则」的逻辑
跑在 AICPU 上更合适。AICPU 本质就是 NPU 上的一颗小 ARM 核，
适合做控制逻辑。

### 8.2 为什么不能在 host 端算？

完全可以，但每次 launch kernel 时都从 host 算再传过去会有延迟。
放在 NPU 上的 AICPU 算，可以全程在设备上完成，减少 host↔device 通信。
对 aclgraph 模式（图模式）尤其友好。

### 8.3 为什么开销公式是 `6M + 10S2`？

这是「Roofline」式的简化模型 —— M 方向和 S2 方向各自代表不同的硬件资源消耗
（M 主要影响寄存器、S2 主要影响 L1 内存带宽）。系数 6 和 10 是工程师实测调出来的。
精度不需要很高，能让 24 个核负载差不多平均就行。

### 8.4 `FA_TOLERANCE_RATIO = 2` 是什么？

容差因子。意思是「分配时可以多塞，只要超出量不超过当前块开销的 1/2」。
这是常见的「First Fit Decreasing」式装箱启发式 —— 宁可前面的核稍多干一点，
也不要后面的核空着。

## 9. 限制与约束（来自 README）

- `num_heads_q` 只支持 64
- `num_heads_kv` 只支持 1
- `cmp_topk` 只支持 512
- `cmp_ratio` 只支持 4 或 128
- `ori_mask_mode` 只支持 4（band）
- `cmp_mask_mode` 只支持 3（rightDownCausal）
- `ori_win_left` 只支持 127
- `ori_win_right` 只支持 0
- `layout_kv` 只支持 `PA_ND`
- 输入 tensor 不能全为 None

这些限制反映出这个算子目前是为了一个**特定模型 / 特定场景**做的特化实现，
而不是通用方案。

> **关于"支持推理场景"的澄清**: 算子 README 写"该接口支持推理场景下使用",
> 容易让人误以为「只能用于推理」。实际语义更准确的说法是:
> 算子只提供**前向**实现, 没有 backward kernel。所以:
> - 推理 (forward only): ✅ 完全支持
> - 训练**前向**调用 (`no_grad`/`inference_mode`/只跑 forward): ✅ 也可以
> - 完整训练循环 (`loss.backward()` 要算 dQ/dK/dV): ❌ 因 backward 缺失而失败
>
> 也就是说限制在"完整训练循环跑不通", 而不是"训练时禁止调用 forward"。
> README 那句话是个不严谨的表述, 不应解读为"仅推理"。
