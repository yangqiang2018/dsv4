为什么"把数据搬进车间"这一步，TileLang 改不动，必须改编译器？

本文用一个 8 行 × 4 列 的小例子，解释我们在昇腾 NPU 上遇到的一个底层限制。不需要任何神经网络或写算子的背景。

一、两个仓库和一条搬运指令

把 NPU 想成一个工厂：

  • GM（全局内存）：城外大仓库。东西多，但离车间远。
  • L1（一级缓存）:车间旁的小仓库。矩阵乘法器（cube）只能用 L1 里的料。

所以每轮计算前都要"GM→L1 搬一批料"。搬运由一条指令完成——TileLang 里写 T.copy(GM区, L1缓冲)，编译器把它翻译成底层模板 copy_gm_to_l1<行数, 列数>(...)。

二、关键：L1 的货架不是按行摆的（NZ 布局）

GM 里数据按行排（第 1 行、第 2 行……）。但 L1 为了喂矩阵乘法器，按"竖条"摆——业内叫 NZ 布局。下图用 8×4 矩阵示意（数字＝第几行）：

重点：一个元素放在 L1 哪个角落，取决于整块的行数。拿第 3 行的 c3 来说：整块 8 行时它在地址 16+4=20，整块 4 行时在 8+4=12——同一个元素、不同的家。这就是后面一切问题的根源。

三、这条指令支持什么、不支持什么

用法支持？说明
整块搬：GM 8行 → L1 8行缓冲✅最常用，没问题
尾块搬：只有 5 行真数据，先清零再搬 5 行✅专为"最后一批不满"设计
中段搬：往 8 行缓冲的第 3~6 行位置补 4 行❌我们需要的正是这个

为什么常用却没人撞坑？因为绝大多数算子一次搬满整块。我们的算子（PageAttention 取 KV）数据按"页"散落在 GM，要分几次往同一个 L1 块的不同段补——立刻踩中。

四、出错过程（8 行块、分两批各 4 行）

先回应一个很自然的疑问：上面 ori_KV[blk, row : row+8, :] 不是已经写了"从第 row 行取 8 行"吗——怎么说不能指定行？

因为一次搬运有两头，"能不能指定行"两头不一样：

  • 源头 GM（按行线性排）："从第几行起、取几行" = "从某地址起、读多长"，天然支持——那个 row : row+8 就是它，没问题。
  • 目标 L1（NZ 竖条排，见第二节）：这条指令只接受一个目标起点地址 + 整块形状，没有"往第几行写"这一项。

把代码摊开就一目了然——能切行段的是源，目标只能给整块：

# ✅ 源(GM)：想取哪几行，随便切
T.copy(ori_KV[blk, row : row+8, :],   # 从第 row 行取 8 行 —— 支持
       kv_l1)                           # 目标：只能整块给

# ❌ 我们真正想要的：只往 L1 的"第 4~8 行"补 4 行
T.copy(ori_KV[blk, row : row+4, :],
       kv_l1[4:8, :])                   # ← 目标也想切行段
#   这行能编过，但 L1 是 NZ：编译器把 kv_l1[4:8] 退化成一个"偏移指针",
#   底层模板仍按"整块从这个指针起"重新铺 NZ + 清零 —— 即下面要演的错。

# 它最终编译成的底层调用，参数表里根本没有"目标起始行":
copy_gm_to_l1<bf16, 8, 4>(   # 8,4 = 整块形状(编译期写死)
    dst_ptr,                # 目标：就一个起点地址(切行段也只是挪它)
    src_ptr, srcN = 4,      # 源：地址 + 每行列数
    tailM = 4, tailN = 4);  # 这次搬几行 / 一行几列有效
#   能调的就这些：块形 + 几行 + 几列 —— 唯独没有"写到目标的第几行"。

所以"指定行"只在源那头成立；到了 NZ 的目标那头，切了也不算数。它对任何一次调用都只有一种理解："这是一个完整 8 行块、从它的开头写起。"下面看它具体错在哪：

而麻烦正好撞上 第二节的 NZ：在 L1 里，一行被拆成两段，分别躺在相隔很远的两个竖条里（如 r1 = 条① 的 a1a2 + 条② 的 a3a4）。下图是这个 8 行块在 L1 的真实样子：

"从第 5 行接着写"意味着：在条① 跳 4 行、同时在条② 也跳 4 行——这是两个不同的地址调整。可指令只收一个起点指针 + 整块形状，一个指针表达不了"在每个竖条里各跳 4 行"。

于是第 2 批只能被指令当成"一个新块从这个指针写起"：它在这个偏移点身后重新铺一套两条，落点和真实的"后半"对不齐；再加上它会把自己以为的"空白行"清零，连第 1 批已写好的另一半也被波及。两批各按各的原点行事、互相打架——实测约一半元素归零或放错（~55% 错）。一次写满整块就没事（只有一个原点、无需偏移），偏偏我们的 KV 来自不同的"页"、必须分批补。

五、为什么 kernel 里绕不过去，要改编译器？

因为"块形"是编译期模板参数：copy_gm_to_l1<8行,4列> 写死在生成的 C++ 里。kernel 作者只能给它一个起点指针，没有任何参数能告诉它"这是大块中间的第 3 行"。下三种绕法都被堵死：

  • 偏移指针 → 被当新块起点（图 2）。
  • 把缓冲拆成 4 个小块 → 矩阵乘法器要整块输入，会多 8 倍调度反而更慢。
  • 每次搬的行数当变量 → 行数也必须是编译期常量。

手写 Ascend C 没这问题：自己直接算 NZ 偏移写进去（约一行代码）。所以这不是硬件不行，是 TileLang 编译器没暴露"块内偏移"这个旋钮。修法约几十行：给拷贝指令加"目标块内行号"参数 + 关掉清零。

六、回到我们的场景

修好之后，搬 KV 的活可由 cube 自己干，省掉一整条"vector 代搬 + 中转仓 + 等待信号"流水（华为官方算子正是如此），相应场景预计还有 2~3 倍加速。在那之前 TileLang 算子只能走中转路，性能上限即当前版本。
