为什么"把数据搬进车间"这一步，TileLang 改不动，必须改编译器？

本文用一个 8 行 × 4 列 的小例子，解释我们在昇腾 NPU 上遇到的一个底层限制。不需要任何神经网络或写算子的背景。

一、两个仓库和一条搬运指令

把 NPU 想成一个工厂：

  • GM（全局内存）：城外大仓库。东西多，但离车间远。
  • L1（一级缓存）:车间旁的小仓库。矩阵乘法器（cube）只能用 L1 里的料。

所以每轮计算前都要"GM→L1 搬一批料"。搬运由一条指令完成——TileLang 里写 T.copy(GM区, L1缓冲)，编译器把它翻译成底层模板 copy_gm_to_l1<行数, 列数>(...)。

二、关键：L1 的货架不是按行摆的（NZ 布局）

GM 里数据按行排（第 1 行、第 2 行……）。但 L1 为了喂矩阵乘法器，按"竖条"摆——业内叫 NZ 布局。下图用 8×4 矩阵示意（数字＝第几行）：

重点：一个元素放在 L1 哪个角落，取决于整块的行数。拿第 3 行的 c3 来说：整块 8 行时它在地址 16+4=20，整块 4 行时在 8+4=12——同一个元素、不同的家。记住这个"按竖条摆、一行拆两段"，下一节要用到。

三、这条指令支持什么、不支持什么

用法支持？说明
整块搬：GM 8行 → L1 8行缓冲✅最常用，没问题
尾块搬：只有 5 行真数据，先清零再搬 5 行✅专为"最后一批不满"设计
中段搬：往 8 行缓冲的第 3~6 行位置补 4 行❌我们需要的正是这个

为什么常用却没人撞坑？因为绝大多数算子一次搬满整块。我们的算子（PageAttention 取 KV）数据按"页"散落在 GM，要分几次往同一个 L1 块的不同段补——立刻踩中。

四、出错过程（8 行块、分两批各 4 行）

先回应一个很自然的疑问：上面 ori_KV[blk, row : row+8, :] 不是已经写了"从第 row 行取 8 行"吗——怎么说不能指定行？

能指定行——这一点你没看错。源头 GM 那侧 row : row+8 是"从某地址读多长"，天然支持；目标 L1 那侧你甚至能写 kv_l1[4:8, :] 让它从第 5 行写起。而且因为 NZ 里每个竖条的步长一致，把目标指针往后挪"4 行"，写入真能正好落进两条的后半（你直觉里"碰巧对齐"的那件事，其实是必然）。所以——写入位置不是坑。

坑在另一个"好心办坏事"的设定：这条搬运指令一次只为"填满一个块"而设计。一旦它发现你这次给的行数比整块少（给 4 行、块却是 8 行），它会先把"它眼里的整块"全部清零，好让你没填的行是干净的 0。从头一次填满，这没问题；可要分两批拼一个块，第 2 批的这个"自动清零"就闯祸了：

# 想分两批填同一个 8 行块：先 r1–r4，再 r5–r8
T.copy(ori_KV[blk, row : row+4, :], kv_l1[0:4, :])   # 第 1 批
T.copy(ori_KV[blk, row+4 : row+8, :], kv_l1[4:8, :]) # 第 2 批：写入能对齐 ✓

# 第 2 批编译成的底层调用：
copy_gm_to_l1<bf16, 8, 4>(dst_ptr, src, srcN=4, tailM=4, tailN=4);
#   tailM=4 < 8(整块) → 模板自动把"从 dst_ptr 起、一整块的长度"先清零，
#   没有任何参数能关掉这步清零 —— 这才是真正的坑。

把整个 L1 缓冲摊成一条地址带（条① 在前、条② 在后），就能看清第 2 批的"清零"扫到了哪里：

于是"分批拼一个块"成了死局：第 2 批为了"填满自己"必先清零整块，而它的清零从偏移点一路扫过第 1 批已写的数据、还冲出缓冲——实测约一半元素被抹（~55% 错）。一次写满整块就没这事（只清一次、清的正好是自己），偏偏我们的 KV 散在不同的"页"、必须分批补。

五、为什么 kernel 里绕不过去，要改编译器？

注意：问题不是"指定不了行"——行能指定，写入也落得对。问题是那步"自动清零"写死在底层模板里、关不掉：模板只要看到"这次的行数 < 整块"，就无条件 InitConstValue 把整块清零。kernel 作者手里没有任何开关能说"我是来补全的，别清零"或"只清我写的这几行"。三条绕法都堵死：

  • 不偏移、从头写 → 第 2 批直接盖掉第 1 批，照样拼不起来。
  • 偏移到第 5 行写 → 写入对了，但那步整块清零照样把第 1 批扫掉（图 2）。
  • 把缓冲拆成多个小块、各自整块填 → 矩阵乘法器要一整块连续输入，拆了要么多搬一遍、要么多算几趟，反而更慢。

手写 Ascend C 没这问题：它的搬运是自己写的循环，想清就清、想补就补。所以这不是硬件不行，是 TileLang 的拷贝模板缺一个"部分填充、不自动清零"的模式。修法在模板里（约几十行）：发现是部分写时，别强制清整块（或只清写入的那一段）。这就是"必须改编译器"的确切含义。

六、回到我们的场景

修好之后，搬 KV 的活可由 cube 自己干，省掉一整条"vector 代搬 + 中转仓 + 等待信号"流水（华为官方算子正是如此），相应场景预计还有 2~3 倍加速。在那之前 TileLang 算子只能走中转路，性能上限即当前版本。
